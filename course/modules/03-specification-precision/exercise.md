# Module 03 Exercise

## Scenario

You receive two requests:

1. Coding: "Add role-based access checks to admin endpoints."
2. Knowledge work: "Prepare a vendor selection recommendation for
   leadership."

Both requests are useful starts but are not execution-ready.

## Task

For each request, produce:

1. Vague baseline (as received)
2. Improved but incomplete spec
3. Executable final spec

Each executable final spec must include:

- objective
- scope
- non-goals
- constraints
- output contract
- acceptance criteria
- validation method
- one approval boundary

## Required Output Format

For each anchor example, submit:

1. Baseline request
2. Improved spec
3. Executable spec
4. Top 3 risks prevented by the final version

## Evaluation Rubric

A strong submission should:

- remove ambiguous language
- define in-scope and out-of-scope clearly
- include measurable acceptance criteria
- include realistic approval boundaries
- keep non-critical details flexible

## Debrief Questions

1. Which field produced the largest quality improvement?
2. What did you keep flexible, and why?
3. How would you detect scope drift during execution?
4. Which anchor was harder to make executable?
